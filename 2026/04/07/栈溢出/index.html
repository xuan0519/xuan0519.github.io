栈溢出

栈溢出

技术名称 目标/作用 实现原理 关键依赖/区别
ret2text 执行程序中已存在的函数或代码片段（例如：一个win()函数）。 覆盖返回地址到目标代码的起始地址。 最简单直接，目标代码必须已在内存中且具有期望的效果。
ret2shellcode 执行攻击者提供的自定义机器代码（Shellcode）。 覆盖返回地址到攻击者注入到栈或其他内存区域的 Shellcode 地址。 需要栈或堆具有可执行权限 (NX 位必须禁用)。
ret2syscall 执行特定的系统调用（如 execve("/bin/sh")）。 覆盖返回地址到一个可以控制寄存器并执行 int 0x80 (Linux) 或其他系统调用指令的 gadget。 不需要注入 Shellcode，但需要找到合适的 gadget 来设置系统调用号和参数。
ret2libc 执行 C 标准库 (libc) 中的函数（通常是 system("/bin/sh") 或 execve）。 覆盖返回地址到目标 libc 函数（如system）的地址，并设置其参数（如字符串 "/bin/sh" 的地址）。 绕过 NX，但需要泄露或知道 libc 的基地址。
ROP (Return-Oriented Programming) 执行任意复杂逻辑，例如多次函数调用或复杂的系统调用。 通过连续覆盖返回地址，将执行流导向一系列短小的、以 ret 结尾的机器指令片段（gadgets）。 最强大的技术，能有效绕过 NX 和 ASLR（结合信息泄露），是现代利用中最常用的手段。

关系

它们是由简到繁、由易到难，并为了克服防御机制而发展起来的技术：

  1. 起点 (ret2text 和 ret2shellcode)： 最早的攻击手段，直接控制 EIP/RIP 到目标代码。
  2. 防御催生 (ret2syscall 和 ret2libc)： 随着 NX (No-Execute) 位的引入，栈变得不可执行，ret2shellcode 失效。攻击者转而利用程序中已有的可执行代码片段（如系统调用指令或 libc 中的函数）。
  3. 终极武器 (ROP)： 随着 ASLR (Address Space Layout Randomization) 和更严格的沙箱机制的引入，仅仅调用一个库函数（ret2libc）可能不够。ROP 通过串联多个 gadget 来构造出完整的、复杂的指令序列，成为绕过所有主流内存保护（NX 和 ASLR）的通用解决方案。

ROP 可以被看作是 ret2libc 和 ret2syscall 的泛化和升级。

  • ret2syscall 是通过少数 gadget 构造一次系统调用。
  • ret2libc 是通过一次返回地址覆盖调用 libc 中的一个函数。
  • ROP 可以用任意多的 gadget 链实现任何想要的逻辑，包括实现多次 ret2syscall 或 ret2libc 调用。

防御机制对比

技术 主要防御机制
ret2text 无（如果目标函数存在）
ret2shellcode NX (No-Execute) 位
ret2syscall ASLR (Address Space Layout Randomization)
ret2libc ASLR (需要泄露 libc 地址)
ROP ASLR (需要泄露地址，并且链的构造更复杂) 和 Stack Canary (但 ROP 针对返回地址，所以 Canaries 主要防范栈溢出，而非 ROP 本身)

漏洞利用场景

1. ret2text (返回到程序中的代码)

  • 场景： 目标程序中有一个理想的函数（例如，一个 win()、shell() 或 exec_shell() 函数）或一段代码，这段代码可以直接授予攻击者想要的权限或执行特定的操作（如打印”You Win!”）。
  • 适用性： 这是最理想、最简单的利用方式，前提是目标函数必须存在且地址固定。
  • 目的： 直接跳过程序正常逻辑，执行预设的成功代码。

2. ret2shellcode (返回到注入的代码)

  • 场景： 攻击者需要执行自定义的指令，例如启动一个反向 Shell、下载恶意文件或执行提权操作。
  • 适用性： 仅适用于目标内存区域（通常是栈或堆）没有开启 NX (No-Execute) 保护机制的旧系统或配置。
  • 目的： 执行攻击者编写的 Shellcode 以获得完全控制权。

3. ret2syscall (返回到系统调用)

  • 场景： 程序开启了 NX 保护，阻止 ret2shellcode，但攻击者只需要执行一个简单的系统调用（如 execve("/bin/sh") 来获取 Shell）。
  • 适用性： 适用于 NX 开启的情况，需要找到一个或少数几个 gadget 来设置系统调用所需的寄存器值。
  • 目的： 绕过 NX，通过操作系统提供的服务（syscall）来实现提权或执行命令。

4. ret2libc (返回到 C 库函数)

  • 场景： NX 保护开启，但程序使用了 C 标准库 (libc)。攻击者想要执行一个强大的库函数，例如 system()、execve() 或 mprotect()。
  • 适用性： 最常见的攻击方式之一，用于绕过 NX。如果 ASLR 开启，则需要先通过信息泄露获取 libc 的基地址。
  • 目的： 利用 libc 中已有的强大函数来获取 Shell (system("/bin/sh")) 或关闭保护机制。

5. ROP (Return-Oriented Programming)

  • 场景：
    • 同时绕过 NX 和 ASLR。
    • 需要执行复杂的逻辑，例如多次函数调用、复杂的内存操作（如分配内存或设置权限）。
    • ret2libc 或 ret2syscall 无法直接满足攻击需求。
  • 适用性： 这是现代漏洞利用中最核心的技术。通过精心构造的 gadget 链，可以实现任意代码执行，是绕过所有主流内存保护的通用解决方案。
  • 目的： 构造一个”虚拟程序”，执行任何攻击者想要的逻辑，以实现复杂的漏洞利用目标。
