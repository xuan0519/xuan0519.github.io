1.Copy File CVE-2026-31431
官方链接:https://copy.fail/
exp
[EXP代码已省略，详见官方链接]
漏洞核心原理
问题来自：
algif_aead
在处理：
splice()
page cache
in-place operation
时的逻辑错误。

攻击者能够：
向只读 page cache 写入受控数据
修改 SUID 程序内存页
不落盘
最终获得 root 权限
这也是它被称为Copy Fail的原因。
危险点
这个漏洞最危险的地方：

  1. 利用稳定
    不需要：
    race condition
    heap spray
    精确 timing
    公开分析称几乎：
    100% reliable
  2. 无文件落盘
    它修改的是：
    page cache
    不是磁盘文件。
    因此：
    AIDE
    Tripwire
    OSSEC
    等文件完整性检测可能无法发现。
  3. 容器逃逸风险
    如果攻击者：
    拿到低权限容器
    拿到 webshell
    拿到 CI/CD 执行权限
    可能直接：
    container -> root
    漏洞利用思路

公开研究显示攻击链大致：
AF_ALG socket
↓
splice()
↓
page cache corruption
↓
修改 SUID 程序
↓
获得 root
例如污染：
/usr/bin/su
的内存页。
CVSS
7.8 HIGH
AV:L/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H
2.Dirty Frag CVE 2026-43284
官方链接：https://github.com/V4bel/dirtyfrag
exp
https://github.com/V4bel/dirtyfrag/blob/master/exp.c
(1900行就不放惹qwq)

（如果我绷住了，我那是不是绷绷炸弹）
漏洞原因
漏洞位于：
xfrm / ESP / IPsec
相关网络子系统。
核心问题：
ESP 解密路径错误处理 shared skb fragments
关键机制
Linux 网络层使用：
skb（socket buffer）
保存网络数据。
其中数据可能来自：
pipe buffer
page cache
splice
zero-copy
这些场景会共享底层物理页。
正常情况下
如果一个 page fragment：
被多个对象共享
内核必须：
Copy-On-Write（COW）
即：
先复制，再修改
否则会污染原始页。
Dirty Frag 的错误

漏洞路径中：
ESP in-place decrypt
直接对共享页：
原地写入（in-place overwrite）
但：
没有正确检查 shared frag 标志
导致：
共享 page 被污染
危险点

  1. 可写 Page Cache
    这是最危险的部分。
    攻击者能够：
    修改 page cache
    而不是磁盘文件本体。
    例如：
    /usr/bin/su
    /usr/bin/passwd
    因为：
    page cache 是执行时真正使用的数据
    即使磁盘文件没变：
    运行中的程序已经被污染
  2. 绕过文件完整性检测
    很多安全工具检查：
    inode
    文件 hash
    磁盘内容
    但：

Dirty Frag 修改的是内存缓存页
因此：
AIDE
Tripwire
RPM 校验
可能完全发现不了。
3. 提权极稳定
不同于传统 kernel exploit：
race
heap feng shui
ROP
Dirty Frag：
利用链非常稳定
因为：
page cache 行为确定性极强
4. 容器风险
如果容器允许：
AF_PACKET
IPsec
相关 socket
则可能：
container → host root
5. 内核级权限
一旦成功：
攻击者通常直接：
UID=0
即：
root

利用思路
这里只讲高层原理。
利用链整体
构造共享 page
↓
splice / MSG_SPLICE_PAGES
↓
生成共享 skb frag
↓
触发 ESP decrypt
↓
in-place overwrite
↓
污染 page cache
↓
覆盖 SUID 程序
↓
执行 SUID
↓
root
第一步：构造共享页
攻击者先：
pipe + splice
让：
page cache 页
进入：
pipe buffer
形成：
shared fragment
第二步：进入 skb
再通过：
MSG_SPLICE_PAGES
把共享页挂入：

skb
第三步：触发 ESP 解密
漏洞路径：
ESP decrypt
错误地：
原地修改共享页
第四步：污染 Page Cache
由于底层页实际上来自：
文件 page cache
于是：
文件缓存被改写
但：
磁盘文件未变
第五步：覆盖 SUID
例如：
/usr/bin/su
被改成：
恶意 shellcode
或者：
setuid shell
第六步：获得 Root
执行：
/usr/bin/su
实际执行的是：

污染后的 page cache
于是：
root shell
CVSS
HIGH
AV:L/AC:H/PR:L/UI:N/S:C/C:H/I:H/A:H
3.Fragnesia CVE 2026-46300
官方链接：https://github.com/v12-security/pocs/tree/main/fragnesia
（这期有点暴力了吧）
exp
https://github.com/v12-security/pocs/blob/main/fragnesia/fragnesia.c
（1200行代码，向坚持手搓的老辈子致敬）
漏洞原因
核心漏洞点在：

  1. ESP-in-TCP / XFRM 处理逻辑错误
    内核在处理：
    ESP 加密/解密路径
    TCP ULP（espintcp）
    skb page fragment
    时发生错误。
  2. page fragment “未正确 unshare”
    关键问题：

共享 page fragment 被错误当成独占页
正常逻辑应该：
shared page → COW（复制）
但漏洞中变成：
shared page → 直接 in-place 写
3. page cache 被错误映射
攻击路径导致：
skb fragment
↓
page cache page
↓
被 ESP 解密路径修改
结果：
只读文件的 cache 被写入
危险点

  1. 本地提权
    攻击者可从普通用户：
    uid=1000 → uid=0
  2. 修改只读文件内容
    可以污染：
    /usr/bin/su
    /bin/passwd
    /lib/libc.so
    systemd 二进制
    这是核心危险点。
  3. 无需复杂竞争条件
    该类漏洞特点：
    不依赖 race
    不依赖 heap feng shui

利用路径固定
exploit 稳定性较高
4. 容器逃逸风险
如果容器允许：
AF_XFRM
ESP socket
IPsec stack
则可能：
container → host root
5. 隐蔽性强
因为修改的是：
page cache（内存）
而不是磁盘文件：
文件 hash 检测可能正常
磁盘扫描看不出变化
利用思路
攻击链通常如下：
Step 1：构造共享 page fragment
使用：
splice
pipe
MSG_SPLICE_PAGES
sendfile
制造：
多个对象共享同一 page
Step 2：触发 skb → page 绑定
将 page fragment 放入：
skb (socket buffer)

Step 3：触发 ESP 解密路径
进入：
xfrm/espintcp
错误逻辑：
in-place decrypt
Step 4：page cache 被污染
导致：
只读文件缓存被修改
Step 5：修改 SUID 程序
例如：
/usr/bin/su → injected payload
Step 6：执行提权
运行：
su
获得：
root shell
CVSS
CVSS v3.1: 7.8 (High)
